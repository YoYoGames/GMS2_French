Ainsi, comme vous pouvez l'imaginer, l'événement Step est vérifié en vérifiant chaque étape du jeu tant que l'instance existe..

Évidemment, lors de la création d’un jeu, il est très important que vous sachiez quand deux (ou plus) occurrences d’un objet sont entrées en collision, et pour cela nous avons l’événement de collision. Il s'agit d'un événement que vous placez dans un objet, puis que vous spécifiez par rapport à un autre objet sur lequel vous devez rechercher des collisions.

Lorsque la physique n'est pas activée, ces collisions seront calculées en fonction du masque des deux objets (le masque est défini dans les propriétés de l'image-objet ou peut être affecté indépendamment dans les propriétés de l'objet) et si elles se chevauchent ou non. Notez que si l'une ou l'autre des instances de la collision n'a pas de masque attribué (ou que le masque d'image-objet est défini sur rien), même s'il en dessine quelque chose, aucune collision ne sera détectée.

Si vous avez activé Physique, la collision sera basée sur le type de forme de collision ( Fixture ) que vous avez défini pour l'objet dans ses propriétés physiques, ainsi que sur sa réaction à la collision. Cela signifie que vous n'aurez peut-être besoin d'aucun code pour traiter la collision, mais cet événement devra toujours contenir au moins un commentaire pour que les collisions soient détectées.

Enfin, il convient de noter que toutes les collisions seront calculées une fois par étape de jeu avant le déclenchement de la collision. Ainsi, lorsque la collision se produit, toutes les collisions auront déjà été calculées et pré-affectées. Cela signifie que si vous créez une instance dans cet événement et essayez ensuite de rechercher une collision avec elle, celle-ci ne sera ni détectée ni résolue jusqu'à la prochaine itération de la boucle de jeu.

Commande d'événement

Lors de la prise en compte d’événements dans GameMaker Studio 2, il convient de noter que l’ordre exact selon lequel les événements vont se dérouler à chaque étape ne peut pas être clairement indiqué, tout simplement parce qu’il dépend du fonctionnement interne de GameMaker Studio 2 et qu’il est susceptible de changer en fonction des modifications. le logiciel se développe. Cependant, certains événements se dérouleront toujours dans le même ordre. Les premiers événements qui se produiront toujours de la même manière sont ceux qui se produisent lors de la première entrée d’une pièce. L'ordre dans lequel les différents événements vont se déclencher est:

  • Variables d'objet / instance Les variables sont initialisées (il s'agit des variables que vous pouvez définir dans l'EDI de l' éditeur d'objet ou de l' éditeur de pièce ). Si vous avez un objet avec des variables attachées, elles seront créées avant toute autre chose, puis toutes les instances associées à des variables seront définies après afin que vous puissiez écraser les variables d'objet par des variables spécifiques à l'éditeur de salle.

  • Evénement Create suivi du code de création d'instance de chaque instance - de sorte que chaque instance est créée, elle commence par exécuter son événement Create, puis son code de création d'instance avant de passer à l'instance suivante à créer.

  • Game Start Event - Cet événement est déclenché une fois dans la toute première salle du jeu pour toutes les instances placées dans la salle à partir de l'éditeur de salle (notez que l'appel game_restart() déclenchera à nouveau cet événement.

  • Code de création de salle - Il s'agit du code unique écrit dans l' éditeur de salle pour la première entrée d'une salle.

  • Événement de début de salle de toutes les instances - l’une des catégories «autres» d’événements et se déclenchera pour toutes les instances, persistantes ou non.


Il convient également de noter que vous pouvez également définir l'ordre dans lequel des instances spécifiques sont créées dans l'éditeur de salle lui-même en les déplaçant vers le haut ou le bas de la liste de la fenêtre Propriétés de la couche d'instance.

Hormis ces événements spécifiques, le seul ordre connu qui se produira toujours de la même manière, quels que soient les événements à trois étapes et les différents événements de tirage au sort. Celles-ci resteront toujours cohérentes, donc si vous avez du code qui repose sur un timing spécifique à chaque étape de votre jeu, vous devez utiliser:

  • Début de l'événement

  • Step Event (notez que l'événement step est exécuté juste avant que les instances ne soient placées dans leurs nouvelles positions)

  • End Step Event


Tous les sous-événements de dessin sont également toujours traités dans le même ordre (à l'exception de l'événement Resize, déclenché différemment):

  • Pré-tirage

  • Tirage au début de l'événement

  • Tirage au sort

  • Fin de tirage

  • Evénement Post Draw

  • Draw GUI Begin Event

  • Dessiner un événement graphique

  • Dessiner un événement de fin d'interface graphique
