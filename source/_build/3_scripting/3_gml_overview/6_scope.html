Variables et portée variable


Comme tout langage de programmation, le langage GML utilise des variables comme unité de base pour la plupart des opérations de programmation. Les variables sont utilisées pour stocker des informations dans la mémoire de l'appareil en vue d'une utilisation ultérieure (ou instantanée). Un nom leur est attribué afin que vous puissiez y faire référence dans des fonctions et des scripts. Une variable en langage GML peut stocker de nombreux types de données différents, comme un nombre réel (comme 100, 2,456575, -56 etc...), une chaîne de caractères (comme "Hello world!"), Un entier (comme 1, 556, -7 ) ou un booléen ( true ou false ).

Une variable est quelque chose que nous utilisons pour stocker une valeur à utiliser dans une ou plusieurs opérations ou fonctions. Pensez à "pi", par exemple... c'est une variable du monde réel qui contient la valeur 3.14159265 (etc...). Pourquoi? Eh bien, il est beaucoup plus facile de dire à quelqu'un "pi" que "trois point un quatre un cinq neuf deux six cinq"! Ainsi, nommer les choses rend la vie beaucoup plus simple et cela signifie aussi que si la valeur de cette variable ne changera jamais, nous ne devons pas changer le numéro partout comme le nom de la variable est toujours le même. En GML, une variable a un nom qui doit commencer par une lettre et ne peut contenir que des lettres, des chiffres et le trait de soulignement «_» d'une longueur maximale de 64 symboles. Donc, les variables valides sont des choses comme fish, foo_bar, num1, et des variables invalides seraient 6fish, foo bar, ou *num.

Maintenant, dans de nombreux langages de programmation, vous devez "déclarer" une variable avant de pouvoir l'utiliser. Cela signifie essentiellement que vous indiquez à l'ordinateur le nom que vous souhaitez utiliser afin qu'il puisse attribuer un emplacement en mémoire pour stocker la valeur que vous décidez de conserver dans cette variable. Avec GML, ce n'est pas toujours nécessaire car cela dépend de la portée de la variable. Lors de la programmation avec GameMaker Studio 2 il existe quatre catégories de variables principales et chacune d’elles a son propre champ d’application (qui peut être considéré comme son domaine d’activité ou sa portée). Une variable d'instance est créée dans une instance d'un objet et est considérée comme unique pour cette instance - c.-à-d. Que plusieurs instances du même objet peuvent avoir la même variable, mais chaque variable peut contenir une valeur différente puisqu'elles sont uniques pour chaque instance. Mais comment créer une variable d'instance? Eh bien, vous créez de nouvelles variables simplement en leur affectant une valeur comme indiqué dans ce petit exemple:

potions = 12;
life = 100;
name = "Jock MacSweeney";
strength = 5.5;
armour = -2;

Comme vous pouvez le constater, il vous suffit de donner le nom, puis une valeur (numérique ou chaîne) pour définir cette variable et l’avoir prête à être utilisée dans une instance de l’objet pour lequel vous codez. Ces variables peuvent ensuite être utilisées et modifiées de plusieurs manières à partir de l'instance, par exemple, ce code pourrait être un événement de collision et être utilisé pour prélever un montant de la variable "life":

life -= 5 + armour;

Si "vie" est à 100, il aura maintenant une valeur de 97 (100 - (5 + -2) = 97). C'est un exemple simple, et vous pouvez remplacer "armure" par la valeur réelle de -2, mais que se passe-t-il si vous utilisez cette valeur à plusieurs endroits et décidez ensuite de la changer? Vous devez passer par TOUT votre code et changer chaque -2 en quelque soit la nouvelle valeur, ce qui prend du temps et est très sujet aux erreurs! Mais si vous utilisez une variable, il vous suffit de lui attribuer une nouvelle valeur. Le code utilisera automatiquement cette nouvelle valeur à partir de ce moment-là, ce qui rendra les choses beaucoup plus souples et plus faciles à corriger en cas de problème. Il convient également de noter que même si une valeur ne va pas changer, il est beaucoup plus facile de se rappeler ce que signifie une variable appelée "vie" plutôt que de simplement regarder un nombre.

GameMaker Studio 2 contient également une collection de variables d’instance "intégrées". Vous devez donc en être conscient, car vous pouvez nommer une de vos propres variables d’instance de la même manière ou si vous souhaitez que votre propre variable globale porte le même nom. obtiennent des erreurs. Cependant, ils sont faciles à repérer, car ils apparaissent dans une couleur différente dans l'éditeur de code et apparaissent également dans la barre de saisie semi-automatique en bas.

Une variable locale est une variable que nous créons uniquement pour un événement spécifique, puis que nous ignorons lorsque l' événement est terminé (la seule exception à cette règle concerne les ressources de script, où var La variable déclarée est locale au script et ensuite rejetée). Pourquoi aurions-nous besoin d'eux? Eh bien, les variables occupent de la place dans la mémoire et il se peut que nous ne les utilisions que pour une opération ou une fonction. Dans ce cas, nous n’avons besoin que de les avoir en mémoire pendant la courte période d’utilisation. Cela permet de garder votre base de code propre et ordonnée, ainsi que de conserver un espace mémoire optimisé pour les éléments qui en ont vraiment besoin. Pour déclarer une variable locale, nous utilisons la fonction var comme ça:

var i, num, str;
i = 0;
num = 24.5;
str = "Hi there!";

Toutes les variables créées ci-dessus seront "oubliées" (c'est-à-dire: supprimées de la mémoire) à la fin de l'événement (ou du script) dans lequel elles ont été créées. Vous devez faire attention à ce que le nom que vous donnez var les variables déclarées ne coïncident pas avec une autre variable d'instance dans l'objet exécutant le code, et assurez-vous également que vous n'avez pas l'intention d'utiliser la valeur stockée dans cette variable en dehors de l'événement, déclarez-le en. Ces variables sont beaucoup utilisées dans les programmes, en particulier dans les boucles pour compter les itérations, ou lorsqu’on utilise une valeur plusieurs fois dans une opération qui ne sera pas répétée. Voici un autre couple d'exemples:

var i = 0;
repeat (10)
   {
   inventory[i] = 0;
   i+=1;
   }

Le code ci-dessus crée une variable locale appelée "i" et la définit sur 0, le tout dans la même ligne. Notez que dans les versions précédentes de GameMaker vous GameMaker d' GameMaker déclarer vos variables locales, puis leur attribuer des valeurs, mais dans cette version, vous pouvez également déclarer et leur attribuer une valeur. Le code ci-dessus utilise ensuite cette variable pour initialiser un tableau. Comme la variable "i" ne va pas être utilisée pour d'autres fonctions dans l'instance autre que celle-ci, elle peut avoir une portée locale. Voici un autre exemple:

var xx,yy;
xx = x - 32 +irandom(64);
yy = y - 32 +irandom(64);
instance_create_layer(xx, yy, "Effects", obj_blood);

Ici, nous avons utilisé les variables locales pour stocker des coordonnées aléatoires que nous utilisons ensuite pour créer une instance. Dans cet exemple, vous pouvez voir qu'il n'est pas strictement nécessaire d'utiliser ces variables, mais dans un souci de lisibilité et de facilité d'utilisation, nous le faisons. Ce que nous faisons là est BEAUCOUP plus clair et évident que si nous utilisions ce code:

instance_create_layer(x - 32 + irandom(64), y - 32 + irandom(64), "Effects", obj_guts);

Une autre chose à propos de var Les variables déclarées doivent être notées... Comme elles sont uniques à l'événement qui les exécute, elles peuvent également être utilisées dans d'autres instances via le code! Cela signifie que nous pouvons utiliser ces variables pour définir et modifier des éléments dans d'autres instances à l'aide de la construction "with ()" (il existe une section à ce sujet dans la section Présentation générale du langage GML du manuel). Le code lui-même ressemblerait à quelque chose comme ceci:

var num = instance_number(obj_Enemy);
with (obj_Enemy)
   {
   if num>10 instance_destroy();
   }

Le code ci-dessus fonctionne parce que le var La variable déclarée est locale à l' événement (ou au script) dans lequel elle est contenue, et non à l'instance, ni au monde du jeu. Elle peut donc être utilisée dans n'importe quelle fonction de n'importe quel objet tant qu'elle se trouve dans le même bloc de code.


Une description de base d'une variable globale est une description qui, une fois déclarée, n'appartient à aucune instance en particulier et reste accessible à tous. Tout comme les variables locales, les variables globales doivent être déclarées, mais contrairement à une variable locale, une variable globale reste en mémoire jusqu'à la fin de la partie. Vous pouvez donc créer une variable globale pour suivre (par exemple) le nombre de puces que possède le joueur, puis simplement mettre à jour cette variable à différents moments de la partie. Une variable globale n'appartient à aucune instance spécifique et peut être consultée, modifiée et utilisée par toutes les instances à tout moment, mais toute modification apportée à la variable est "globale", dans la mesure où toutes les instances utilisant la variable seront affectées par la modification.. Regardons un exemple:

global.food = 5;

Nous déclarons la variable "food" en écrivant d'abord "global" puis un "." pour indiquer à GameMaker Studio 2 que cette variable est maintenant une portée globale. Nous devrons désormais utiliser cette méthode chaque fois que nous aurons besoin d'accéder ou de modifier cette variable de quelque manière que ce soit. Nous avons donc créé une nouvelle variable appelée "nourriture" et nous l'avons déclarée globale. Désormais, toute instance peut utiliser et modifier cette variable de quelque manière que ce soit et toutes les autres instances verront cela. Par exemple, nous pourrions avoir un objet alimentaire différent avec lequel le joueur entre en collision et dans l'événement de collision, nous avons:

global.food +=1;

Nous avons aussi un autre objet qui dessine cette valeur comme ceci:

draw_text(32, 32, "food = " + string(global.food));

Avec les variables globales, nous pouvons modifier les valeurs et voir ces modifications reflétées dans toutes les instances des objets faisant référence à cette variable. Comme pour les variables locales, vous devez veiller à ne pas nommer vos variables globales de la même manière que toutes les variables d'instance, car cela pourrait vous causer des problèmes et faire en sorte que des bugs se glissent dans vos jeux en raison du chevauchement des variables, ce qui peut parfois être un problème difficile à déboguer. En général, vous devez avoir un seul script ou un seul objet qui déclare toutes vos variables globales au tout début du jeu (par exemple, dans l’événement de démarrage de la salle), de sorte qu’elles soient initialisées et prêtes à être utilisées par toute instance. vous donne un endroit pratique pour revenir en arrière si vous devez vérifier un nom de variable.

GameMaker Studio 2 contient également une collection de variables globales "intégrées". Vous devez donc en être conscient, car vous pouvez nommer l'une de vos variables d'instance de la même manière ou si vous souhaitez que votre propre variable globale porte le même nom et vous demandez pourquoi vous êtes. obtenir des erreurs! Cependant, ils sont faciles à repérer, car ils apparaissent dans une couleur différente dans l'éditeur de code et apparaissent également dans la barre de saisie semi-automatique en bas. La majorité des variables globales intégrées sont très spécifiques et ne seront utilisées que rarement - et sont répertoriées dans les sections appropriées du manuel - mais il en existe une importante qui est utilisée fréquemment et ne figure pas ailleurs:

La méthode suivante peut également être utilisée pour déclarer des variables globales, mais elle n'est incluse que pour des raisons de compatibilité, et il n'est pas recommandé de l'utiliser pour les nouveaux projets, car les versions futures de GameMaker Studio 2 risquent de ne pas la prendre en charge.

La deuxième méthode pour créer des variables globales consiste à les déclarer comme telles en utilisant globalvar déclaration, tout comme une variable locale utilisant le var déclaration.

IMPORTANT! le globalvar déclaration est obsolète et pris en charge uniquement à des fins d’héritage. Vous devriez toujours utiliser le global. identifiant pour marquer les variables globales.


Cette déclaration (obsolète) serait utilisée comme suit:

globalvar food;
food = 5;

Une fois déclarée de cette manière, la variable "aliment" est maintenant considérée comme globale et ne nécessite aucune modification. global. prefix - ce qui signifie également qu'il est beaucoup plus difficile d'identifier les variables globales dans votre code et qu'il est également beaucoup plus facile d'obtenir un chevauchement de variables lorsque vous utilisez le même nom de variable dans différents objets ou à partir d'extensions que vous avez installées. Une fois déclarée de cette manière, la variable globale est accessible comme suit:

food += 2;

ou:

draw_text(32, 32, "food = " + string(food));

Comme vous pouvez le constater, si rien n’indique que la variable ait une portée globale, vous vous préparez potentiellement à faire face à de nombreux problèmes subtils dans votre jeu. C’est pourquoi cette déclaration doit être évitée.

Bien qu'elles ne soient pas exactement des variables, les macros sont similaires à leur utilisation, c'est-à-dire qu'elles sont des valeurs nommées que vous pouvez utiliser dans votre code pour remplacer des valeurs absolues. Fondamentalement, une macro (également appelée constante lorsqu'elle est utilisée de la sorte) est une variable nommée qui contient une valeur unique constante (ou chaîne). Vous pouvez définir vos propres constantes à l'aide de l' éditeur de script, puis les utiliser dans votre code et DnD comme s'il s'agissait de variables normales, à la différence qu'elles ne peuvent pas être modifiées dans le jeu. Par exemple, supposons que vous définissiez la macro suivante comme une constante (notez le "#" précédent et l'absence de deux-points ";" à la fin):

#macro total_weapons 10

Vous appelleriez alors ceci dans votre code comme ceci:

if ++pos == total_weapons
   {
   pos = 0;
   }

Notez que vous ne pourrez pas changer la valeur constante, donc un code comme celui-ci fera planter le jeu:

total_weapons = 11;

Vous pouvez définir une macro n'importe où dans votre code ou dans vos scripts. Elle sera pré-compilée et incluse dans votre jeu comme si elle existait déjà, mais nous vous recommandons de créer une ressource de script dédiée et de définir toutes vos macros. Il sera plus facile d'organiser et de déboguer plus tard!

Si vous avez besoin que la valeur d'une macro change au moment de l'exécution, vous devriez probablement en faire une variable globale, puisqu'elles peuvent être modifiées à partir du code pendant un jeu, à moins que vous ne définissiez la macro comme une fonction. En définissant la macro sur une fonction, cela signifie que cette fonction sera appelée à chaque fois que vous utilisez la macro. Par exemple:

#macro col make_colour_hsv(irandom(255), 255, 255)

Vous appelleriez alors cette macro quelque chose comme ceci:

image_blend = col;

En utilisant ce code, l’image mélangera une couleur différente à chaque utilisation de la macro. Il est à noter que vous pouvez également fractionner des macros sur plusieurs lignes à l’aide de la barre oblique. \ pour montrer où la ligne se rompt. Un exemple serait quelque chose comme:

#macro hello show_debug_message("Hello" + \
string(player_name) + \
", how are you today?");

C’est purement esthétique, car le fractionnement d’une telle macro n’aura aucun effet sur le résultat de la dernière macro et consistera simplement à prendre en charge le texte multiligne sur des macros comportant des lignes de code plus longues.

Une caractéristique très importante des macros est qu’elles peuvent être définies pour être utilisées avec des configurations spécifiques (configs), ce qui signifie que vous pouvez avoir le même nom de macro mais lui attribuer des valeurs différentes en fonction de la configuration actuellement sélectionnée. Par exemple, supposons que vous avez une configuration pour Android Ads et une autre pour iOS Ads, vous pouvez alors définir une seule macro pour contenir la valeur requise de l'ID d'application:

#macro ad_id "";
#macro Android:ad_id "com.yoyogames.googlegame"
#macro iOS:ad_id "com.yoyogames.appstoregame"

Comme vous pouvez le voir, vous donnez le nom de la configuration en premier, puis deux points: puis le nom de la macro et sa valeur. Notez que vous ne pouvez avoir aucun espace entre les deux points: et le nom de la configuration, ni le nom de la macro, sinon vous obtiendrez une erreur.
