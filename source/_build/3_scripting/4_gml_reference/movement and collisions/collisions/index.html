Vérification de collision


Lors de la planification de mouvements ou de la décision de certaines actions, il est souvent important de déterminer s'il existe des collisions avec d'autres objets à certains endroits du monde du jeu. Choisir la bonne collision pour le travail est souvent la tâche la plus importante. GameMaker Studio 2 comporte un certain nombre de fonctions intégrées qui vous aideront à gérer les collisions correctement et conformément aux besoins de votre projet.

IMPORTANT! Toutes les fonctions de collision utilisent des positions entières, quelles que soient les valeurs entrées. Toute valeur non entière sera arrondie à l'entier le plus proche.


Les fonctions suivantes traitent des différentes manières dont une instance peut vérifier les collisions avec une autre instance:

  • place_empty
  • place_free
  • lieu_mise
  • position_empty
  • position_meeting
  • changement de position
  • position_destroy
  • instance_place
  • instance_place_list
  • instance_position
  • instance_position_list

Les routines suivantes peuvent être utilisées à cet effet (ainsi que d'autres fonctions détaillées dans les sections relatives au déplacement et aux instances ). Ces fonctions de collision fonctionnent toutes de manière légèrement différente, mais elles maintiennent trois arguments communs que nous expliquerons ici:

  • obj. C'est ici que nous spécifions l'objet avec lequel la fonction doit rechercher une collision. Il peut s'agir d'un identifiant d'instance, d'un type d'objet ou du mot clé spécial GM all. Il convient également de noter que si le type d'objet choisi est le parent désigné d'autres objets, tous les enfants de cet objet seront également inclus dans cette vérification.


  • préc. Cela peut être soit true ou false et se rapporte à la précision de la vérification des collisions. Si réglé à true, alors des collisions précises doivent être calculées (mais seulement si l’objet recherché est marqué "précisément" pour sa détection de collision dans ses propriétés de masque), ce qui est une tâche légèrement plus complexe pour la CPU et doit donc être utilisé avec prudence. Si false les collisions seront alors calculées en fonction de la boîte englobante de l'objet à vérifier.


  • notme. Vous devrez peut-être vérifier les collisions avec un objet ayant le même index que celui qui exécute le code. Par conséquent, pour éviter que le code ne détecte une collision avec lui-même, vous pouvez le définir sur true. Si vous recherchez spécifiquement l'instance de l'objet exécutant le code, ou si cela n'a pas d'importance, vous pouvez le laisser comme suit: false.

Les fonctions de base renverront soit l’ID d’une instance trouvée en conflit, soit le mot clé spécial noone lorsqu'il n'y a pas de collision, les fonctions de liste renverront le nombre d'instances en conflit et rempliront une liste DS prédéfinie avec leurs ID. Notez que s'il existe plusieurs collisions avec les zones définies par ces fonctions et instances de l'objet donné, un seul ID d'instance est renvoyé et il peut s'agir de l'une des instances de la collision.

Les fonctions suivantes existent qui traitent des collisions avancées.

  • cercle_collection
  • liste_circle_collection
  • collision_ellipse
  • collision_ellipse_list
  • ligne de collision
  • liste_de_collections
  • point de collision
  • collision_point_list
  • collision_rectangle
  • collision_rectangle_list

Toutes les fonctions ci-dessus sont liées aux instances de contrôle de collision et reposent donc sur le masque de collision défini pour l'instance. Cependant, il existe de nombreux moments où vous devez vérifier les "collisions" avec un point ou une zone, en particulier lorsque votre instance ne possède pas d'image-objet assignée, ou lorsque vous GameMaker la souris, etc.... Par conséquent, GameMaker Studio 2 fournit les fonctions suivantes pour vous aider dans ces situations:

  • point_in_rectangle
  • point_en_triangle
  • point_dans_circle
  • rectangle_en_rectangle
  • rectangle_en_triangle
  • rectangle_dans_circle

Lors de l'utilisation de la physique intégrée, il n'est pas garanti que les fonctions de collision ci-dessus fonctionnent pour les instances activées par la physique. Ceci est principalement dû au fait que ces instances n'utilisent plus la majorité des variables intégrées standard (à la place, les instances activées par la physique ont leur propre ensemble de variables ) et n'utilisent pas non plus le masque de collision ou le cadre de sélection, car elles utilisent des fixtures. au lieu. Cependant, il existe une fonction spéciale pour tester les collisions avec la physique activée qui vérifie le chevauchement des fixtures:

  • physics_test_overlap
